# 第二课：从 demo 跨过"真能用"的门槛

> 上一课停在"agent 能跑了"。但**能跑 ≠ 能用**——入口散、概念混淆、没评测、session 数据跟着仓库走。这课把这些问题一个个挖出来、一个个处理。
>
> 假设你能读代码、能查文档、能问模型。重点是为什么，不是怎么。

---

## 引子

01.md 收尾时，agent 还是 demo 形态：`python agent.py "..."` 跑一行 stdout。**好玩，但不"能用"**。从这一课开始，处理的是**demo → 真实可用**之间必须解决的事。

这一课里很多 commit 不是新功能，而是**"把已有的东西摆正"**。最关键的一个决定（session 该放哪）甚至翻转了之前定下的判断。

---

## 阶段 1：跑通之后，几个 immediate 的小清理

### 删 calc 工具（5d4502d）

之前 `agent.py` 里有个 `calc` 工具：用 `eval()` 实现，**清了 `__builtins__` 试图安全**。01.md 的 docstring 自己也写了警告——"通过 `__class__` / `__mro__` 的 gadget chain 仍可能漏，**不要用于不可信输入**"。

"demo 级安全" → "真能用"的临界点上，**这种已知漏洞不能再留**。直接删。剩下三个工具：`bash` / `read_file` / `current_time`。

不是"优化"，是**收尾**。

### 一键启动脚本（2739d0e）

`./codeagent` 启动 TUI：57 行 bash，自定位脚本目录、依赖检查、首次自动 `npm install`、彩色 banner、透传 args。

**为什么这玩意儿也算"真能用"的一部分**？因为入口复杂度直接决定别人会不会试。`npm start` 多了三步（`cd` 到仓库、`npm install`、`npm start`），**多数路人卡在第一步**。

> ★ **关键洞察："能不能用"很多时候不在功能在入口。**
>
> 功能可以烂一点，**入口简单**用户会原谅。入口复杂，**功能再好**用户到不了。

这不是核心功能，但**真正决定"用得起来用不起来"**。这类工作容易被低估。

## 阶段 2：术语对齐——我们存的不是"memory"

这一阶段做的事**只是一个 rename**，但背后是一个**概念澄清**。

之前 `.codeagent/memory.sql` 里存的是**每轮请求/响应的 raw payload**。`8c0dbd2` 把它改成 `session.sql`。

**为什么？** 我们讨论 ChatGPT Memory 功能时意识到：**他们说的"memory"是从 session 里抽出来的事实**，不是 session 本身。**严格的术语里：**

- **session** = 原始对话轨迹（含工具调用、中间结果、模型解释）
- **memory** = 从 session 抽出的"事实"（"用户偏好 X"、"项目 Y 在做 Z"）

我们的 `.codeagent/...` 里存的是前者。**叫 memory 容易让人误以为它已经是"事实抽取后的产物"**。rename 让概念对齐。

> ★ **关键洞察：文件名是文档。**
>
> 同一个文件叫 `memory.sql` 还是 `session.sql`，**下游工程师的预期完全不同**。一个会期望里面有"用户偏好"，一个会知道里面是 raw trace。

这一阶段没改一行逻辑。**改的是命名**。但它影响未来——如果未来真要做"从 session 抽取 facts"的功能，**叫 session 的文件就是源数据，叫 memory 的文件就是产物**，不能再混淆。

## 阶段 3：把散在根目录的文件包化（9948e37）

仓库一开始是平铺的：`chat.py` / `agent.py` / `contexts/` / `tui/` / `tui.jsx`。规模不大，**但"谁是谁"不一眼看清**。

`src/code_agent/` 包化做了这件事：

```
src/code_agent/
├── llm/         chat.py, chat.sh       模型访问层
├── exec_loop/   agent.py               推理 + 工具执行
├── contexts/    base.py, memory.py     system prompt 注入
└── tui/         tui.jsx + step demos   终端 view 层
```

**为什么是 4 层不是 6 层或 2 层**？**刚好对应一条链**："input 进来 → 模型决策 → 上下文注入 → view 输出"。每个目录一个职责，**新人 30 秒能找到文件**。

不是先验设计。**是跑了 30 次之后发现"我找个工具函数翻了 5 个文件"才动的**。

实现细节有两个不显眼的点：

- `sys.path` bootstrap 让 `from code_agent.X import Y` 不管从哪个 cwd 都能解析
- `MEMORY_DIR` **这一阶段还是 `Path.cwd() / '.codeagent'`**——下一个阶段才改

这个细节本身就是这一课的伏笔：**当时还不知道 session 该放哪**。

## 阶段 4：评测 harness（19e334c）

**为什么造一个 eval 框架**？很简单：**没有评测就没法迭代**。

改 agent 改得心里没底——"现在比之前好了吗？"——只能凭感觉。评测框架把"agent 干得好不好"变成可量化的指标：

```
eval/
├── run_eval.py             主循环：per-task 隔离 workdir → 跑 agent → 喂 grader
└── tasks/
    ├── registry.py         drop a .py with TASK/GRADER_CODE，加进 _load
    ├── sort_integers       write_code 类（纯算法）
    ├── count_log_lines     bash_task 类（扫 /tmp 下的 .log）
    └── copy_and_modify     multi_step 类（读 → 改 → 校验）
```

**三类 task 是怎么选的**？**对应 agent 在真实开发里最高频的三种动作**：
- 写代码（写一个排序函数）
- 调 bash（在日志里数行数）
- 多步组合（读一个文件，复制并改一处，再校验改对没）

每类一个 task。**不是题量大，是"模式覆盖"**。smoke test 三个全过（100%）。

> ★ **关键洞察：评测不是为了证明 agent 能干活，是为了把"干得好不好"从感觉变成数字。**
>
> 这条之前在 01.md 的"没做"清单里出现过。这一课做了**最简版**——三道题不是完整评测集，但是**可以往里加新题的脚手架**已经搭起来了。**可扩展性**才是这个阶段的关键产出。

## 阶段 5：session 该放哪——一个回旋（0372357）

**这是这一课最重要的一次决定**——也是**翻转了之前判断**的一次。

### 之前的判断

01.md 收尾时我们刚把 `.codeagent/memory.sql` 跟 git 一起推了（b8c41fe）。理由："个人练手项目，session 数据就是文档的一部分"。`tutorial-generator` skill 还把 `.codeagent/memory.sql` 当成数据源之一。

**这个判断后来被推翻了**。`0372357` 是直接的产物——`MEMORY_DIR = Path.home() / '.codeagent'`，session 数据从 work dir 搬到 home 目录，**自动迁移 + gitignore flip**。

### 为什么改主意

三个理由汇到一起：

**1. 产品逻辑的迁移**。如果你做这个 agent 准备给别人用，session **绝不能跟着用户的 git 走**。每个用户有自己的会话，**不该污染别人的 repo**。

**2. 行业惯例**。Claude/Codex/Aider 全都把 session 放 home——`~/.claude/sessions/`、`~/.codex/sessions/`、`~/.config/aider/`。**这是定型的设计**，跟着走没错。

**3. 持久化考量**。session 跟着 work dir 走，**意味着用户换机器 = 数据全丢**。放 home，至少跟着用户的 home 备份策略（TimeMachine / rsync 等）走。

> ★ **关键洞察：session 是"用户的工作记忆"，不属于"项目代码"。**
>
> 同一份代码在多台机器 clone、多个 repo 共用同一个 agent——session 应该是**用户粒度**的，不是**项目粒度**的。
>
> 01.md 当时的反向操作有它的理由（个人项目 + tutorial skill 要读），但**不能成为产品形态**。

### 迁移本身的设计

`_migrate_legacy_memory()` 启动时检查——

- 老的 cwd 内 `.codeagent/` 还在的话挪到 home
- **不覆盖已有 home 数据**（保护可能已经存在的旧 session）
- 写一个 `.migrated_from_cwd` flag 标记，避免重复迁移

**这种"自动一次性迁移"在用户升级时**特别重要——**不能让人手动 cp 文件**。这是工程上的"对用户友好"，不是技术上的"难"。

### 这件事的认识论意义

`0372357` 这个 commit **直接诞生于对话讨论**。我们在对话里挖到"session 该放 home 不在 work dir" → 落地成 `MEMORY_DIR = Path.home() / .codeagent`。

**这是"讨论改变代码"的典型例子**。这种 commit 之后会越来越多，因为我们对 session/memory 的认识还在演化。

## 阶段 6：聊到的几个概念，没直接成 commit 但塑造了决定

这一课里好几次讨论**没直接产出 commit**，但影响了上面的设计。

### WAL + fsync

讨论"session 丢失怎么办"时挖出来的两个机制。**SQLite WAL 是我们已经在用的防线**——`.codeagent/session.sql-wal` 兜底进程崩溃。`synchronous=FULL`（默认）保证每次提交都 fsync，**最多丢"还没提交的当前事务"**。

```
WAL:    解决"SQLite 文件结构不会被破坏"
fsync:  解决"写入主文件后，那个内容真的在盘上"
```

两层叠加 = **进程崩 + 断电的常见场景都扛得住**。**云同步我们没做**——个人项目，离商业化还远。

### "WAL / JSONL / Kafka / Git" 是同一个形状

`先 append-only 到一个文件 / 后台周期合并` 这套模式在**很多层都出现**。它的精髓不是"出错概率降低"，是**"出错半径缩小"**——单点失败不再级联成全局失败。

这个抽象当时让 0372357 的"session 在 home"变得更踏实：home 目录的本地存储 + WAL **已经够防常见故障**，云同步是以后的事。

### 自然语言怎么挖规律

聊到"agent 跑出来的样本能不能改进 agent 自己"——结论是**不能纯靠自然语言**，但 agent 样本里**工具调用是结构化的**，所以比纯文本容易挖。SFT、RL with outcome reward、self-reflection、process mining——**这些是"零用户数据"的训练路径**，整个行业都在摸索。

**这条认知影响下一个未知阶段**——当我们真的要给 agent 做 fine-tuning 时，从 session 抽 synthetic agent traces 走的是这条路。

## 横向总结

| 我们做了 | 为什么 |
|---|---|
| 删除 calc 工具 | 已知 gadget-chain 漏洞 |
| codeagent 一键启动 | 入口复杂度决定用不用得起来 |
| rename `memory.*` → `session.*` | 严格区分 session（raw trace）vs memory（curated facts） |
| `src/code_agent/` 包化 | 4 层职责清晰，新人能秒定位 |
| eval harness（3 task 类型） | 把"干得好不好"从感觉变数字 |
| session 搬 `~/.codeagent/` + 自动迁移 | 数据粒度 = 用户粒度，不该绑在 repo |

| 我们没做 | 为什么 |
|---|---|
| **完整评测集** | 3 题只是脚手架；评测是连续投入 |
| **跨 session 的 fact 抽取** | "memory" 这层还停在概念；session → memory 流水线没建 |
| **云同步** | home 目录 + WAL 已防常见故障；云同步是后期事 |
| **多用户 / 团队场景** | 产品形态没确定 |
| **跨 session 真正自动的 memory pool** | "session / memory" 这两层架构还在演化 |
| **Anthropic / OpenAI 兼容的 session 格式** | 自己定义的格式够用；兼容是商业化时的事 |
