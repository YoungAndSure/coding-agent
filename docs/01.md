# 第一课：搭起一个 Coding Agent

> 这一篇是**开发过程笔记**，不是技术手册。
> 重点是为什么，不是怎么。
> 假设你能读代码、能查文档、能问模型。

---

## 引子

目标一开始很小：让一段脚本跟 LLM 说上话。然后一路上，每个新问题都把目标推得更远一点。

本课记录的是**问题链**——什么问题冒出来，我们做了什么决定，为什么这么做。不是教程，**是路线图**。后面谁（人也好 agent 也好）拿到这份 doc，应该能接着往下走。

---

## 第一阶段：让脚本跟 LLM 说话

最开始的需求：**让脚本读 `~/.claude/settings.json` 里 `ANTHROPIC_BASE_URL` / `AUTH_TOKEN` / `MODEL`，调通接口**。半小时能搞定。

**为什么写两个版本？** Python 版（`chat.py`，用官方 SDK）和 shell 版（`chat.sh`，纯 curl）。**不是因为需要两个**，是因为写完 Python 版之后脑子里冒出一个问题：**SDK 到底干了什么**？是接到了一个黑盒，还是一堆能看穿的活儿？

把 SDK 拆开看，就是 6 件事：拼 URL、加 header、序列化、错误转异常、httpx 发请求、反序列化。**没有任何"魔法"**。第二版用 shell 跑 curl 是为了把这个判断落地——"用 curl 能跑到的事，SDK 也能跑到"。

> ★ **关键洞察：所谓"接入 LLM"，没有任何一个动作是不可替换的。**
>
> 这个判断比想象中重要。后面所有复杂能力（流式、工具调用、sub-agent）都是在这个"普通 HTTP + JSON"基础上搭的。看清这一层，agent 世界的复杂度被剥掉一大半。

## 第二阶段：流式输出

跑 `chat.sh` 的第一反应是：**模型整段生成完才显示**。能不能边生成边打印？

答案是**改一个字段**。请求体加 `"stream": true`，服务端从"返一个 JSON"切到"SSE（Server-Sent Events）"——一个 event 一行，行里包 `delta.text`，客户端把这些增量拼起来写到 stdout。

这部分有两个反复推敲的点：

**粒度**。起初的直觉是"每个 token 一个事件"，但实际看 SSE 事件，**通常一个 content_block_delta 里塞几个 token**（视服务端 batching 而定）。这个粒度对终端显示够用了，体感是"打字机"。

**append-only**。每个 `delta.text` 不会跟前面的重合、不会覆盖——客户端逻辑就是 `text += delta.text`，**不需要 dedup、不需要 diff**。这个保证来自模型的"自回归"性质：上一个 token 落定之后就不变了。

> ★ **关键洞察：`stream: true` 不是个"美化输出"选项——它切的是协议本身。**
>
> `false` → 一个 JSON。`true` → SSE。**同一个 URL，两种协议**。这个视角的切换在后面 TUI 那段会反复用到。

这个阶段没做的：流式高亮、双向流式输入。这些等真正撞上痛点再说。

## 第三阶段：让 LLM 自己决定要不要调工具

把模型能力从"讲话"推进到"做事"的那一步，是工具调用协议。

**为什么是这个时机**？跑流式时发现：模型能讲，但只能讲。让它真做事，得加 tool-use。同时下一步要做的 TUI 会一直接着打印"模型在干嘛"——**没有工具调用，UI 那一截就没法动**。

工具调用协议一上来就遇到两个反直觉：

**第一个**：**每个请求都得带 tools 数组**。服务端不记得你上次传过啥。所以"先注册一次，之后只传 id"这种想法行不通。

**第二个**：模型返回的可能不是纯文本，而是 **`text` 块和 `tool_use` 块混在一起**。模型可以一边说"我先看看文件"一边调 `bash`。这不是错误，是协议故意这么设计的。

`agent.py` 就是围着这两条事实搭的：

```python
messages = [user_prompt]
while True:
    resp = send(messages + tools)
    blocks = parse(resp)
    tool_calls = [b for b in blocks if b.type == "tool_use"]
    messages.append(assistant's blocks)        # 必须保留原结构
    if not tool_calls: break
    results = [run(t) for t in tool_calls]     # 一轮所有
    messages.append(user tool_results)          # 一次回灌所有
```

> ★ **关键洞察：所有 agent 都跑这同一个 5 行循环。**
>
> Claude Code、Cursor Agent Mode、Aider、Devin……不管外面长什么样，骨头都是这五行。**差别全在它周围那一圈配套**（权限、UI、监控、记忆……）——我们后面会一段一段撞上去。

**协议硬约束**：回灌时 `tool_use_id` 必须配对到前一条 assistant 消息里的 `tool_use.id`，否则 400 报错。**所以循环里那段 `messages.append(assistant's blocks)` 不可省**——少了它，下一轮 model 看到 tool_result 时找不到它问过的问题，整个 context 就断了。

### 工具是宽还是窄

先做了一个 `bash` 工具（让模型自己 `ls` `cat` `git`），没拆成 `Read` `Grep` `Glob` 一堆窄工具。三个原因：

1. 协议层先跑通，再去抠工具设计
2. 想看模型对环境的反应——`command not found` 这种错误本身就是有价值的反馈
3. 实际工程里拆不拆，看你跑两轮后的判断

> ★ **关键洞察：工具是"宽还是窄"是个可调参数，不是先验。**
>
> 第一版你不知道模型会用得多准。跑两轮你就能看出哪些 bash 调用其实可以拆。**这个调优本身是一个跟模型共同演化的过程**。

### 工具名字 vs 工具实现

模型对"Grep"这个名字有强先验（训练数据里见过海量），但**真约束是 description + input_schema**。

换个名字叫 `search_files` 模型照样能学会，但描述得补全。"Grep 是怎么实现的"——是系统命令？`ripgrep`？还是纯 Python `re`？**模型根本不知道你背后是什么**。你写啥都行，schema 是协议，name 是提示符。

## 第四阶段：让用户"看到"这一切

第三阶段跑出来只有 stderr 流水和 stdout 一段文字。**TUI 的需求从这里冒出来**——能不能让"模型在干嘛"实时显示？

这个阶段被故意拆成**四个子步骤**，每个都有自己的探问点。**拆得越细越讲得清楚**。

### 4.1 先确认最底层的字节流

`tui_step1.py` ~20 行：切 raw 模式，阻塞读 1 字节，打印 codepoint。这是在验证"raw 模式到底有没有切对"。任何抽象框架都会包住这一层，先确认最底层的行为，后面对任何 TUI 库都不会迷茫。

### 4.2 对比 line 模式

`tui_step1_line.py` 不切 raw，按 `input()` 读整行。这一步纯粹是为了分清"raw vs line"两种工作模式——绝大多数工具是 line 模式（Python REPL、bash、git、docker……），TUI 才需要 raw。这个对照一旦建立，所有 CLI 工具的"按键响应速度"差异都能用这套解释。

### 4.3 自己写个最简 TUI

`tui_demo.py` ~50 行，全标准库。手动拼 ANSI 串（`\x1b[2J\x1b[H` 清屏、`\x1b[K` 清行、`\x1b[36m...\x1b[0m` 颜色……），手写主循环 `while: render → read key → update state`。

> **为什么一定要自己先写一遍才上框架**？不写就**不懂库在帮你挡什么**。写完才发现，**TUI 没有组件**——所有看起来像 GUI 控件的"组件"在终端里都是字符拼接。ANSI 协议本身有规律：每条以 `0x1B` 起、以一个字母（m/J/K/H）结尾。

> ★ **关键洞察：TUI = 字节流 + 状态机 + 主循环。**
>
> 全屏界面底层就是"不停把 ANSI 串写到 stdout + 不停从 stdin 读 1 个字节"。**没有 DOM，没有 canvas，没有任何渲染图元**。

### 4.4 然后才上框架

`tui.jsx` + `package.json`。装 `ink` / `react` / `ink-text-input`，用 `tsx` 转译 JSX。

Ink 本质是"把 ANSI 拼接封装成 React 组件"。`<Box flexDirection="column">` 不是盒子——是"Ink 帮你算了哪些 ANSI 跳格指令该拼起来"。**你写 React state，Ink 帮你拼 ANSI、帮你 diff、帮你管 raw 模式**。如果不先看 4.3，Ink 看起来就像在变魔法。

JSX 不是 TypeScript。四个后缀（`js` / `jsx` / `ts` / `tsx`）是不同组合：JS vs TS × 有没有 JSX 标签。`jsx` 是不带类型的"JS + XML-in-JS"语法糖。

**架构上**：TUI 进程 spawn `agent.py` 子进程，每次回车起一个独立 Python 跑完拿 stdout。**两个进程，TUI 只管 view，agent.py 只管干活**。这是后续 stream 的钩子——目前是"一次性发回"，每个 token 流过来重画才是真体验。

## 第五阶段：让它别失忆

到第四阶段末尾，**有个洞**被发现了。

跑了几次 `python agent.py "..."` 之后，问题浮出来："我发送一条消息时，LLM 返回若干 tool，我执行完再把结果返回，但是返回时候我没收集 context"——**单次 loop 不丢，但跨 invocation 丢**。下次 `python agent.py` 又是一段全新对话，**金鱼 agent**。

**这个洞揭示了一个边界**：协议层保证了**单次 invocation 内** messages[] 全保留、tool_use ↔ tool_result 配对，**上下文不断**。但**跨 invocation 这一刻，state 全部销毁**。

要补这个洞，服务端不帮——它从不记得你传过啥。**只能客户端自己存**。

`agent.py` 加了一层：
- `.codeagent/memory.json`：每轮请求/响应 append 一行（人可读、可 `git diff`）
- `.codeagent/memory.sql`：同样的事用 SQLite 写（按 iter/kind 高效过滤）
- `contexts/base.py`：`ContextBuilder` 接口，`title` + `build() -> str`
- `contexts/memory.py`：`RecentConversationsContext` 查最近 10 轮对话
- `build_contexts()`：每次发请求前把所有 context 拼起来，**加到 system prompt 头部**

> ★ **关键洞察：协议不变，记忆靠我们这边自己存。**
>
> 跨 invocation 共享的不是 messages[]，**是 .codeagent/ 里的持久化数据 + 加载到 system prompt 的那一段文本**。两条不同的链路。

**为什么双写 json + sql**：
- JSON：人可读，能 `cat` 能 diff，git 历史好看
- SQL：可查询，能按 iter/kind 高效过滤
- 看起来冗余，但**两边都 fail-soft**（任一写失败不阻塞主流程，stderr 一行警告）

> ★ **关键洞察：辅助存储必须是 fail-soft 的。**
>
> 主流程跑通了，记忆系统是附加价值。**最差的那一面先被牺牲**——对话能用，记忆丢几条，用户体感比"对话直接报错"好一截。

`ContextBuilder` 抽象是为了**让加新 context 类型零成本**。今天可以加 `ProjectDocsContext("docs/")`、明天加 `UserPreferencesContext()`，每一类一段独立文本，拼到 system prompt。

## 横向总结：做过 / 没做 + 理由

| 我们做了 | 为什么 |
|---|---|
| SDK 版 + curl 版 chat 并行 | 验证 SDK 是包装，不是魔法 |
| 流式从协议层手动实现 | 体会"SSE 是另一个协议" |
| 一个笼统的 bash 工具 | 协议层先跑通；后续再拆 |
| TUI 子步骤拆 4 段 | 框架是"省"工程的，不能反过来"挡"学习 |
| 记忆 = 本地 + system prompt 注入 | 协议不变，记忆只能客户端解 |

| 我们没做 | 为什么没做 |
|---|---|
| **Web UI** | 跟 TUI 是不同 surface；TUI 这条线还没走通 |
| **sub-agent** | 概念层聊过、协议层没动；agent 还太简单 |
| **沙箱 / 写操作确认** | 还在 demo，模型输出我们还信得过；上线前补 |
| **Evaluation 套件** | 任何"agent 干得好不好"的判定标准本课里都没有 |
| **Cost 跟踪** | 还没在跑量场景 |
| **TUI 流式化**（每个 token 重画） | 当前是 spawn 一次性拿；下一阶段自然会撞上 |
| **Tool 拆分** | 跑两轮再看 |

### 一句话带走

> **复杂系统的代码短，是因为"智能"在别处。**
>
> loop 5 行、toolspec 50 行、TUI 80 行。
