# 第一课：搭起一个 Coding Agent

> 这一篇是**开发过程笔记**，不是技术手册。重点是 **为什么** 而不是 **怎么**。技术细节假定你能查到或模型知道。
>
> 时间范围：几次迭代，从空白目录到能跑起来的 TUI agent。

---

## 引子

出发时只有一个目标：**让一段脚本跟 LLM 通上话**。这目标在半小时内达成，但**走到最后远不止这目标**。过程中我们问了十几个问题，每个问题都决定了下一步的方向。本课不重复答案里已经清楚的部分（比如"怎么用 SDK"、"JSON 怎么解析"），只记录**我们怎么走到这里的、为什么这里走、为什么不走那条路**。

读者画像：
- 知道 Python、懂点 HTTP
- 没用过"agent"框架，但好奇
- 想把"调用 LLM"的认知推到更深一层

最后会有一个**横向的"我们做过 / 我们没做 + 理由"**清单，作为给后续课（`02.md`、`03.md`...）的引子。

## 第一阶段：让脚本跟 LLM 说上话

### 做了什么

写了两个等价物：`chat.py`（用官方 `anthropic` SDK）和 `chat.sh`（用 `curl`）。两者读的是同一份 `~/.claude/settings.json`，调的是同一个 endpoint。

### 为什么从这开始

最朴素的需求：让"我打一句话" → 程序打回一句话。**先打通主链路**比先考虑结构重要。SDK 和 curl 两个版本同时写，是因为写完想回答一个问题——**SDK 到底干了什么**？

### 这一步产生的认知

写完两个版本后发现，**没有 SDK 这一层，业务也能跑**。SDK 在做的事只有 6 件：

1. 拼 URL（`base_url + /v1/messages`）
2. 加 header（`x-api-key`、`anthropic-version`）
3. 序列化和反序列化
4. 错误码转异常
5. 打字打到终端（……并没有）
6. **没了**

> ★ **关键洞察：所谓"接入 LLM"，没有任何一个动作是不可替换的。**
> 
> 后面所有的复杂能力（流式、工具调用、sub-agent）都是在这个"普通 HTTP + JSON"基础上搭的。**理解了这一点，整个 agent 世界的复杂度都被剥掉了一层**。

## 第二阶段：让 LLM 边想边说

### 做了什么

在 `chat.sh` 加了一个 `--stream` 开关，向请求体加 `"stream": true`。然后写了一段 Python 内嵌脚本（`grep '^data: '` + `sed` + `json.loads`）来解析 SSE 流。

### 为什么会想到这

第一次跑 `chat.sh` 时发现：模型整段回复**全部生成完才显示**。下面要做的工具调用也依赖"能看到正在发生什么"，所以"流式"是个**立刻有需求**的能力。

> ★ **关键洞察：`stream: true` 不是个"输出美化"选项——它切的是协议本身。**
> 
> `false` 是一次 HTTP 请求 → 一个完整 JSON。  
> `true` 是同一个 URL → 内容变成 SSE（Server-Sent Events），每个事件一行，事件结束看 `delta.text`。
> 
> 大模型训练资料里 SSE 写得清楚，所以细节不在这里赘述。一个判断这个阶段是否完成的标志：**你能心算出为什么客户端必须用 `curl --no-buffer`**。

### 这个阶段我们没做的事 & 理由

- **没做"流式高亮"**（边说边给代码块加 markdown 标记）—— 等真正遇到痛点再说。
- **没改成双向流式输入** —— 流式只解决了"输出"，"输入"暂时一条 prompt 一次够了。

## 第三阶段：让 LLM 自己决定要不要调工具

### 做了什么

写了 `agent.py`，实现一个完整的 tool-use loop：

```
messages = [user_prompt]
while True:
    resp = send(messages + tools)
    blocks = parse(resp)
    tool_calls = [b for b in blocks if b.type == "tool_use"]
    messages.append(assistant's blocks)
    if not tool_calls: break                # 没活儿干，结束
    results = [run(t) for t in tool_calls]
    messages.append(user tool_results)      # 一次发回所有
```

注册了 4 个工具：`bash`、`read_file`、`calc`、`current_time`。

### 为什么是在这个时刻

第二阶段跑流式时我们发现：**模型能讲话，但只能讲话**。要让它真做事，得加 tool-use 协议。同时，下一步要做的 TUI 会一直接着打印"模型在干嘛"，离不开"工具调用"这一截。

> ★ **关键洞察：所有 agent 都跑这同一个 5 行循环。**
> 
> Claude Code、Cursor 的 Agent Mode、Aider、Devin……不管外部长什么样，骨头都是这五行。差别不在 loop，而在它周围那一圈配套（权限、UI、监控、记忆……）。

### 这一刻为什么选"一个 bash 工具"而不是"一堆窄工具"

我们讨论过这个问题。表层看，**窄工具更安全**：

- `Read`、`Grep`、`Glob` 都不通过 shell，天然无沙箱
- 结构化输出（不是裸 stdout）
- 自动跳过二进制 / 限结果数

我们**还是先做了一个 `bash`**。原因有三：

1. **demo 阶段**：5 行能跑起来 > 严格的权限划分
2. **想看模型对环境的反馈**：bash 让模型能看到"git 未安装"这种错误，从而试错
3. **后面再拆**：先把协议跑通，再去抠工具设计

> ★ **关键洞察：工具是"宽还是窄"是个可调参数，不是先验。**
> 
> 写第一版时你不知道模型会用得有多准确；跑两轮后你就能看出哪些 `bash` 调用其实可以拆。这个调优本身是一个**跟模型共同演化的过程**。

### 协议里最容易踩的一个坑

最初我写的循环里，先 batch 处理所有 tool_use、再一次性回灌 —— 这是协议强约束。但**为什么是"批量"而不是"处理完第一个就发回去"**？模型本身也允许一对一处理。

原因是 **`tool_use_id` 必须有 `tool_result` 配对**。如果只回灌其中一个，剩下的 `tool_use_id` 就"对不上"了，模型连那条消息都看不到，context 断了。

> ★ **关键洞察：协议不是"建议"，是"硬约束"。**
> 
> 你发给 LLM 的消息必须按 `assistant(tool_use) → user(tool_result)` 顺序配对，**否则在协议层就直接 400 报错，根本到不了模型**。这是为什么所有 agent 实现都长一个样。

## 第四阶段：让用户能"看到"这一切

这一阶段被拆成了**四个子步骤**，因为"在终端画东西"这个需求拆得越细越讲得清楚。

### 4.1 先理解终端

写 [`tui_step1.py`](../tui_step1.py)：切 raw 模式，一个按键阻塞读 1 个字节。打印出按键的字符和 codepoint。

**为什么先写这个**：我们想验证"raw 模式到底有没有切对"、"按键到底以什么字节到达"。**任何抽象框架都会包住这层，先确认最底层的行为，后面读任何 TUI 库都不会迷茫**。

### 4.2 对比 line 模式

写 [`tui_step1_line.py`](../tui_step1_line.py)：关掉 raw 模式，按 `input()` 读整行。

**为什么要有这个对照**：Claude Code 看起来很 fancy，但 `chat.py` 的 REPL 模式其实就是 line 模式。**普通 CLI、Python REPL、bash 都是 line 模式**。绝大多数工具根本不需要 raw，TUI 才需要。所以"raw vs line"是先搞清楚的一个分叉。

### 4.3 自己写一个最简 TUI

写 [`tui_demo.py`](../tui_demo.py)：~50 行，全标准库。手动拼 ANSI 串（`\x1b[2J\x1b[H` 清屏、`\x1b[K` 清行、`\x1b[36m...\x1b[0m` 颜色...），手写主循环 `while: render → read key → update state`。

**为什么一定要自己先写一遍才上框架**：

- 不写就**不懂库在帮你挡什么**
- 写完才发现，**TUI 没有组件** —— 所有 fancy 的"组件"在终端里都是字符拼接
- ANSI 协议本身有规律：每条以 `0x1B` 起、以一个**字母**（m/J/K/H）结尾

> ★ **关键洞察：TUI = 字节流 + 状态机 + 主循环。**
> 
> 看起来像 GUI 的全屏界面，底层是"不停把 ANSI 串写到 stdout + 不停从 stdin 读 1 个字节"。**没有 DOM，没有 canvas，没有任何渲染图元**，只是字符级的协议。

### 4.4 然后才上框架：Ink

写 [`tui.jsx`](../tui.jsx) + `package.json`。装 `ink`、`react`、`ink-text-input`，用 `tsx` 转译 JSX。

**为什么最后才上 Ink**：

- 上面 3 步是"我懂我用的工具"的过程
- Ink 的价值是**让你不用自己算布局、拼 ANSI、做 diff**——但如果你不知道它替你挡了什么，你就**没法 debug**
- 真实工程里，agent 的 TUI 几乎都用 Ink/Textual 这些框架，**但能写出`tui_demo.py`级别 demo 的人才懂那些框架为什么存在**

> ★ **关键洞察：Ink 本质是"把 ANSI 拼接封装成 React 组件"。**
> 
> `<Box flexDirection="column">` 不是"盒子"，是"Ink 帮你算了哪些 ANSI 跳格指令该拼起来"。React 的 state → render 循环在终端里跑出来就是"按键 → 整屏重画"。**就是这么朴素**。

### 这一阶段的几个反复的小问题

中途有几个**第一次没理解清楚**的问题，对它们的多次推敲最终才让 TUI 这块在脑子里成型：

- `JSX ≠ TypeScript`，四个后缀（`js/jsx/ts/tsx`）分别是什么意思
- Bash 行模式 vs raw 模式——我们亲手写了两版才知道差别
- `Bash('xxx')` 里的字符串进 shell 前端还要先拼字符串——这就是 TUI 跟 command 的关系

## 第五阶段：让它别失忆

### 做了什么

加了一个 `.codeagent/` 目录 + `contexts/` 模块：

- `.codeagent/memory.json`：每轮请求/响应 append 一行
- `.codeagent/memory.sql`：同样的事用 SQLite，便于查询
- `contexts/base.py`：定义 `ContextBuilder` 接口（`title` + `build() -> str`）
- `contexts/memory.py`：`RecentConversationsContext`，查最近 10 轮对话
- `agent.py` 里：每次发请求前 `build_contexts()` 把所有 context 拼起来 → 加到 system prompt 头部

### 为什么会想到这

到第四阶段末尾跑了几次 `python agent.py "..."`，每次都是一个新进程。**这次跑完到下次开始中间没有任何连续性**——提问上下文、聊过的功能、上次决定的偏好，**全部没了**。

讨论里有两句话把这个洞点出来：

> "返回 tool_result 时没收集 context，下一步该干什么 LLM 不知道。"
> "单次 loop 不丢，但跨 invocation 丢。"

### 这一刻的关键判断

> ★ **关键洞察：协议不变，记忆靠我们这边自己存。**
> 
> 服务端不记得我们传过啥，记忆只能本地解决。**两次 invocation 之间共享的，不是 messages[]，是 .codeagent/ 里的持久化数据 + 加载到 system prompt 的那一段文本**。

我们设计了 `ContextBuilder` 抽象（`title + build()`），是为了**让加新 context 类型零成本**：

```python
DEFAULT_CONTEXTS = [
    RecentConversationsContext(limit=10),
    # 后续可以加：
    # ProjectDocsContext("docs/"),
    # UserPreferencesContext(),
    # CodingStyleContext(".codeagent/style.md"),
]
```

### 为什么双写 json + sql

当时讨论里提过这个：

- **JSON** 可读、可 diff、能 `cat`
- **SQL** 可查询、按 iter/kind 高效过滤

看起来冗余，但**两边都能 fail-soft**（任何一个写失败不阻塞主流程，stderr 记一行警告）。这是设计原则。

> ★ **关键洞察：任何辅助存储必须是 fail-soft 的。**
> 
> 主流程跑通了，记忆系统是"附加价值"。如果记忆写失败阻塞对话，agent 就废了——用户感受最差的那一面先被牺牲。

## 横向总结：做过 / 没做 + 理由

下面这张表是本课的"路线图留白"。后续 `docs/02.md`、`03.md`... 会分别从这里挑一两块展开。

| 我们做了 | 为什么 |
|---|---|
| SDK 版 + curl 版 chat 并行 | 验证 SDK 是包装，不是魔法 |
| 流式从协议层手动实现 | 体验"SSE 是另一个协议"这件事 |
| 一个笼统的 bash 工具 | demo 优先；后面想清再拆 |
| 子步骤拆 TUI（step1 → line → demo → jsx） | 框架是"省"工程的，不能反过来"挡"学习 |
| 记忆 = 本地 + system prompt 注入 | 协议不变，记忆只能客户端解 |

| 我们没做 | 为什么本课不做 |
|---|---|
| **Web UI** | 不同 surface，TUI 这条线还没走通不该并行 |
| **sub-agent** | 概念层聊过，协议层没动；agent 还太简单不需要拆 |
| **沙箱 / 写操作确认** | 还在 demo，模型输出我们还信得过；上线前才补 |
| **Evaluation 套件** | 任何"agent 干得好不好"的判定标准本课里都没有；这是 `02.md` 的主题 |
| **Cost 跟踪** | 不花自己的钱；上线跑量时才有意义 |
| **Tool 拆分** | 上面表里说了，先糊后精 |
| **持久化 messages 到 jsonl** | 跟 memory.sql 重叠，留着下次改写时一并做 |

## 写在最后

### 这一课最重要的"骨架"

如果只让你记一句话，从这次开发里带走：

> **复杂系统的代码短，是因为"智能"在别处。**
>
> 我们的 loop 5 行、toolspec ~50 行、TUI ~80 行。
>
> 真正撑起来的"智能"——
> - 模型自己选的 tool_use
> - 模型决定的 reasoning
> - 模型生成的 text
>
> 都不在我们的代码里。
